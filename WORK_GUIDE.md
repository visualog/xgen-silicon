# xGen 작업 가이드

## 목적
xGen은 노드 기반 인터페이스로 브랜드 일러스트 생성 워크플로를 구성하는 Next.js 16 애플리케이션입니다. 사용자는 프롬프트, 스타일, 화면 비율, 해상도를 조합하고, 영문 생성 프롬프트를 확인한 뒤 이미지를 생성할 수 있습니다. 백엔드 파이프라인은 Gemini 기반 생성을 우선 시도하고, 필요하면 Pollinations로 폴백합니다.

## 프로젝트 구조
- `src/app/page.tsx`: 메인 React Flow 화면과 클라이언트 상태 흐름
- `src/components/nodes/*`: 프롬프트, 스타일, 비율, 해상도, 출력, 캔버스 노드 UI
- `src/app/api/translate/route.ts`: 프롬프트 번역과 미리보기용 영문 프롬프트 조합
- `src/app/api/analyze-style/route.ts`: 업로드 이미지 스타일 분석
- `src/app/api/generate/route.ts`: 최종 이미지 생성 파이프라인
- `src/lib/ai-provider.ts`: Gemini CLI 연동, 프롬프트 빌드, 해상도 계산, Nano Banana 생성 헬퍼

## 핵심 화면 흐름
1. 메인 화면에는 `Prompt`, `Style`, `Ratio`, `Resolution`, `Output` 노드가 기본 배치되어 있습니다.
2. 사용자는 각 노드의 값을 입력하고, 엣지를 연결하거나 변경해 어떤 입력이 출력 노드로 들어갈지 결정합니다.
3. 관련 입력값이 바뀌면 앱이 디바운스 후 영문 프롬프트 미리보기를 요청합니다.
4. 생성이 시작되면 필요 시 `Canvas` 노드가 동적으로 추가되고 `Output` 노드와 연결됩니다.
5. 반환된 이미지는 캔버스 노드에 표시되며, 선택한 비율과 해상도, 마지막 결과 이미지는 로컬 스토리지에 저장됩니다.

## API 흐름

### `/api/translate`
- 입력값: `prompt`, `style`, `ratio`, `resolution`
- 역할: 생성 전 미리보기용 `englishPrompt` 생성
- 규칙: 일반 설명 프롬프트는 Gemini로 번역·강화하고, 스타일 문장은 사용자가 작성한 내용을 그대로 유지합니다.
- 특이사항: 설명 프롬프트 없이 스타일만 있는 경우 비율과 해상도 태그를 함께 붙입니다.

### `/api/analyze-style`
- 입력값: `imageBase64`, 선택적 `mimeType`
- 역할: 업로드한 참고 이미지의 시각 스타일을 분석해 재사용 가능한 영문 스타일 프롬프트 반환
- 구현 방식: 임시 파일로 저장한 뒤 Gemini CLI에 `@파일경로` 형태로 전달하고, 분석 후 파일을 삭제합니다.

### `/api/generate`
- 입력값: `prompt`, `style`, `ratio`, `resolution`, 선택적 `prebuiltPrompt`
- 역할: 최종 생성 프롬프트를 조합하고 base64 데이터 URL 이미지를 반환
- 규칙: `prebuiltPrompt`가 있으면 재사용해 Gemini 프롬프트 빌드를 중복 호출하지 않습니다.
- 폴백 순서:
  1. Nano Banana를 통한 Gemini 네이티브 이미지 생성 시도
  2. 실패 시 Pollinations.ai로 폴백

## 로컬 상태와 UX 규칙
- 로컬 스토리지 키: `brandgen_state`
- 저장 값: `lastImageUrl`, `ratio`, `resolution`, `theme`
- 이 앱은 엣지 연결 여부를 기능 토글처럼 사용합니다. 노드에 값이 있어도 `output-node`에 연결되지 않으면 실제 출력에 반영되지 않아야 합니다.
- `Output` 노드는 연결된 프롬프트에 텍스트가 있거나, 연결된 스타일이 활성화되어 있을 때만 생성 가능 상태가 됩니다.

## 실행 전제와 런타임 가정
- 의존성 설치: `npm install`
- 로컬 개발 서버 실행: `npm run dev`
- `.env.local`에 `GEMINI_API_KEY` 필요
- 백엔드 실행을 위해 Gemini CLI 사용 가능해야 함
- `src/lib/ai-provider.ts`는 현재 프롬프트 빌드 경로에서 `/usr/local/bin/gemini`를 가정함
- 스타일 분석 라우트는 `gemini` 명령어 이름으로 실행하므로, CLI 경로 일관성을 확인해야 함

## 안전한 수정 체크리스트

### 메인 화면을 수정할 때
- 노드 ID를 바꿀 경우 연결 로직과 저장된 상태 복원 흐름까지 함께 수정할 것
- 번역용 `useEffect`의 의존성 배열을 건드릴 때는 요청 루프가 생기지 않는지 확인할 것
- 연결된 입력과 연결되지 않은 입력을 구분하는 로직을 유지할 것
- 생성된 이미지 상태가 계속 `CanvasNode`로 전달되는지 확인할 것

### API 라우트를 수정할 때
- `page.tsx`의 요청/응답 형태와 서버 응답 스키마를 맞출 것
- `prebuiltPrompt`가 있을 때 프롬프트 빌드가 다시 일어나지 않도록 유지할 것
- 외부 모델 호출의 타임아웃과 에러 처리를 보존할 것
- 폴백 동작은 부가 기능이 아니라 제품 계약의 일부로 취급할 것

### Gemini 연동을 수정할 때
- stdout 파싱은 JSON 응답과 일반 텍스트 응답 모두를 견딜 수 있게 유지할 것
- CLI 출력이 항상 완벽한 구조라고 가정하지 말 것
- 라우트별로 절대 경로와 명령어 경로 사용 방식이 일관한지 다시 확인할 것

## 권장 검증 절차
1. `npm run dev`로 앱을 실행합니다.
2. 메인 화면에서 기본 노드 그래프가 정상 렌더링되는지 확인합니다.
3. 프롬프트를 입력하고 `/api/translate`를 통해 영문 미리보기가 갱신되는지 확인합니다.
4. 스타일을 추가하거나 선택한 뒤, 해당 문장이 재해석되지 않고 미리보기에 반영되는지 확인합니다.
5. 생성 실행 후 캔버스 노드가 나타나고 이미지가 표시되는지 확인합니다.
6. 페이지를 새로고침한 뒤 마지막 이미지, 비율, 해상도, 테마가 복원되는지 확인합니다.

## 현재 구현 특성
- `README.md`는 아직 기본 Next.js 템플릿 중심이라 실제 워크플로를 설명하지 못합니다.
- 이 프로젝트의 실제 제품 표면은 일반 폼이 아니라 React Flow 기반 그래프 UI입니다.
- 백엔드는 사실상 Gemini CLI를 내부 서비스처럼 사용합니다.
- 이미지 생성 성공 여부는 외부 모델 상태와 폴백 경로 가용성에 영향을 받습니다.

## 기능 확장 시 원칙
- 새 노드 타입은 기존의 엣지 기반 입력 모델에 자연스럽게 들어갈 수 있을 때만 추가할 것
- 프롬프트 미리보기 로직과 최종 이미지 생성 로직은 분리된 상태를 유지할 것
- 새 환경 변수가 필요해지면 즉시 문서에 반영할 것
- 새 백엔드 기능이 사용자 흐름을 바꾸면 이 가이드와 `README.md`를 함께 업데이트할 것
